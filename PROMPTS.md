# Каркас промптів: де що використовується і як керує роутером та моделлю

> Карта ВСІХ промптів/правил системи: що зашито (механіка двигуна) і що
> редагується (база/панель). Конфіг тенанта вивантажується/завантажується одним
> файлом: Налаштування → «Конфіг одним файлом» (експорт/імпорт `tenant_config.json`).

---

## 1. Дві фази на КОЖНЕ повідомлення

```
Повідомлення клієнта
   │
   ▼
ФАЗА A — РОУТЕР (модель вирішує наступний крок)
   системний промпт роутера =
     ROUTER_PROTOCOL (МЕХАНІКА, код)        ← формат JSON-дії, цикл
     + {decision_rules} (РЕДАГОВАНЕ)        ← як діяти, куди по дані
     + [WHO YOU ARE] коротка персона
     + [BUSINESS RULES]
     + [CHAT MEMORY] / [ALREADY CHECKED] / [GATHERED FACTS]
   → модель повертає JSON: {"action","query","reason","memory_patch"}
   │
   ▼  (виконується інструмент, результат → у факти; цикл повторюється)
   │
   ▼
ФАЗА B — ВІДПОВІДЬ (модель пише клієнту)
   системний промпт відповіді =
     persona (system_prompt, РЕДАГОВАНЕ)    ← тон, особистість
     + [BUSINESS RULES] + [MARKETING]
     + [GATHERED FACTS] + tpl_evaluation_rules + escalation_prompt
     + price-gate (якщо не питали ціну)
     + answer_style (РЕДАГОВАНЕ)            ← як писати: живо, з гумором
   → жива відповідь українською → _clean_answer (зрізає службове сміття)
```

## 2. Що ЗАШИТО в коді (механіка — не редагується)

| Що | Файл | Чому в коді |
|---|---|---|
| Формат JSON-дії `{action,query,reason,memory_patch}` | `agent.py` ROUTER_PROTOCOL | синтаксис двигуна |
| Цикл «рішення → інструмент → збір → відповідь», ліміт кроків, антидубль | `agent.py` run_agent | механіка |
| Перелік дій (генерується з enabled_tools) | `agent.py` TOOL_DESCRIPTIONS | прив'язка до коду інструментів |
| JSON-режим роутера (response_format) | `agent.py` / `llm.py` | технічне |
| Запобіжники GUARD (коли форсувати) | `agent.py` | логіка циклу; ТЕКСТИ/тригери винесені в БД |

## 3. Що РЕДАГУЄТЬСЯ (база `bot_settings`, панель)

### Колонки таблиці
| Поле | Призначення |
|---|---|
| `system_prompt` | **персона/тон** (хто ти, як звучиш) |
| `business_rules` | жорсткі заборони бізнесу |
| `marketing_rules` | апсел/крос-сейл |
| `escalation_prompt` | що казати, коли факту нема |
| `temperature` / `max_tokens` / `llm_model` | параметри моделі |

### Ключі `meta` (JSONB)
| Ключ | Призначення |
|---|---|
| `engine` | `agent` / `classic` |
| `agent_max_iterations` | макс. кроків циклу |
| `enabled_tools` | які інструменти увімкнені |
| `agent_decision_rules` | **ПРАВИЛА РІШЕНЬ** — як діяти, куди по дані |
| `answer_style` | **СТИЛЬ ВІДПОВІДІ** — як писати клієнту |
| `parts_instruction` | логіка/трактування пошуку запчастин |
| `parts_sites` | сайти запчастин (стороння ринкова ціна) |
| `fallback_sites` | довірені сайти (наш сайт перший) |
| `serper_api_key` | ключ Google-пошуку (НЕ експортується) |
| `tpl_evaluation_rules` | анти-галюцинації |
| `price_triggers` | слова → шукати ціну |
| `capability_triggers` | слова → перевірити каталог («чи робите») |
| `business_info_triggers` | слова → get_business_info |
| `brand_words` / `part_words` | для ринкового пошуку деталей |
| `catalog_synonyms` | синоніми (екран=матриця) |
| `business_info` | контакти, години, оплата, доставка |

### Інші таблиці (теж редаговані)
- `service_categories` / `service_prices` — прайси
- `qa_pairs` — Q&A, `kb_documents` — документи (+ Qdrant RAG)
- `knowledge_types` — Схема Логіки (інтенти + reasoning-шаблони)
- `channels` — Telegram-бот / userbot / webchat

## 4. Інструменти (роути), куди агент іде по дані

| Дія | Джерело даних | Редагується через |
|---|---|---|
| `list_categories` / `search_catalog` | наш прайс (SQL) | Прайси + синоніми/стоп-слова |
| `search_knowledge` | QA + документи (Qdrant) | База знань |
| `search_parts` | сайти запчастин → Google | parts_sites + parts_instruction |
| `web_research` | Google(Serper)/DDG + парсинг | fallback_sites + serper_api_key |
| `open_url` | конкретна сторінка | — |
| `get_business_info` | business_info | вкладка Бізнес-факти |
| `escalate` | оператор | (заглушка) |

## 5. Як підтягнути конфіг через інтерфейс

1. **Експорт:** Налаштування → «⬇ Експорт усіх промптів/правил» → отримуєш `tenant_config.json` з усіма полями цього тенанта.
2. **Імпорт:** Налаштування → вибрати файл → «⬆ Імпорт» → усі поля заповнюються з файлу.
3. Готовий приклад-каркас із поточними даними — `tenant_config_template.json` (у репо). Його можна імпортувати в новий тенант і потім підправити.

**Що НЕ в файлі:** `serper_api_key` (секрет) і механіка двигуна (вона в коді).
