# Universal Engine Cleanup — 2026-06-19

Цей файл описує локальні зміни після переходу від старого action-loop бота до універсального LLM-driven engine для різних бізнесів: сервіс, магазин, інші тенанти.

## Головна ідея

Поточна архітектура:

1. **Chat model** бачить головну персону тенанта, коротку історію, очищені факти route-ів і формує відповідь клієнту.
2. **Controller model** окремим викликом вирішує, чи треба відкривати route. Вона не пише клієнту й не вирішує бізнес-факти сама.
3. **Route model** окремою короткою памʼяттю працює тільки зі своїм джерелом: каталог, Q&A, бізнес-інфо, веб, зовнішні ціни, оператор.
4. У chat model повертаються тільки очищені факти/умови/стан, а не сирі бази.

Ключове правило: якщо бізнес-факт не підтверджений route-ом, модель не має вигадувати “так/ні/ціна/адреса”.

## Активні основні файли

- `app/core/agent_lean.py` — головний runtime нового двигуна: conduct, controller, route session, tool call, final answer.
- `app/core/route_tools.py` — інструменти для внутрішніх баз: каталог, категорії, Q&A, business_info.
- `app/core/prompt_defaults.py` — дефолтні промпти для нових тенантів і міграцій. Runtime читає промпти з БД, а не хардкодить поведінку.
- `app/givi_system_prompt.md` — системна персона тестового тенанта texno.plus / Інженер Андрон.
- `app/admin/routes.py` — веб-панель: налаштування, routes, каталог, Q&A, business_info, імпорт/експорт.
- `app/admin/templates/settings.html` — налаштування моделі, conduct/ban, промптів.
- `app/admin/templates/knowledge/logic_form.html` — редагування route-ів і їх промптів.
- `app/admin/templates/knowledge/prices.html` та `price_form.html` — категорії, послуги/товари, ціни й універсальні поля.
- `app/core/tools.py` — веб-пошук, парсинг URL, зовнішні джерела. Веб-пошук не видалявся.

## Route-и нової логіки

- `catalog` — що бізнес продає/ремонтує/надає, категорії, товари/послуги, власні ціни.
- `qa` — правила, процеси, умови, Q&A, документи.
- `business_info` — адреса, графік, телефон, оплата, доставка, контакти.
- `web_search` — визначити незрозумілий тип предмета або зовнішню характеристику.
- `external_price` — зовнішні ціни запчастин/комплектуючих/ринкових пропозицій.
- `handoff` — передача оператору або контактна інструкція.

## Каталог

Каталог працює у два рівні:

1. **Категорії**: короткий список заголовків і описів. Для питання “чи робите X?” route бачить тільки категорії, без цін і рядків послуг.
2. **Рядки товарів/послуг**: назва, тип, ціна/умова, наявність/статус, характеристики, що входить у роботу/комплект, примітка для моделі.

Ціни називаються тільки якщо є відповідний рядок прайсу для того самого subject/service/operation. Якщо є лише широка категорія, але немає конкретної ціни — цифра не називається.

## Що виправлено 2026-06-19 пізнім патчем

- Якщо controller повертає порожньо на короткий follow-up типу “а телефони??”, fallback відкриває `catalog`, а не віддає відповідь без перевірки.
- Слово “телефон” більше не веде саме по собі в `business_info`; для контактного телефону потрібні слова типу “номер”, “контакти”, “ваш телефон”, “подзвонити”.
- Для price follow-up типу “так хоч орієнтовно ціну” fallback бере недавній user-контекст, щоб route шукав “телефон / Samsung / заміна екрана”, а не голе “орієнтовно”.
- Якщо route model не змогла сформувати query, engine робить короткий fallback-query зі структурованого route request, але відповідь усе одно має підтвердити route validator.
- Conduct більше не відповідає загальним fallback-ом на лайку. Якщо warning LLM мовчить, повертається коротке попередження.
- При `conduct_warnings = 2`: перша пряма образа = попередження, друга = бан сесії.
- `list_categories` показує тільки активні категорії.

## Веб-пошук

Веб-пошук лишився навмисно:

- `web_search` — коли треба зрозуміти, що це за невідомий предмет/модель.
- `external_price` / `search_parts` — коли потрібна зовнішня ціна конкретної запчастини/комплектуючої.
- `open_url` — для конкретного сайту або URL-шаблону.

Веб не підтверджує, що бізнес це ремонтує/продає, і не є власним прайсом бізнесу.

## Як читати проблему в логах

Поганий симптом:

- `DECIDE` порожній.
- route не відкрився.
- `ANSWER` каже “у базі не бачу підтвердження”.

Правильна поведінка після патчу:

- На фактичне питання про предмет/послугу fallback відкриває `catalog`.
- Якщо category match підтверджує предмет — chat model відповідає згідно з route facts.
- Якщо category match не підтверджує — тільки тоді chat model каже, що послуга/товар не підтверджені.
