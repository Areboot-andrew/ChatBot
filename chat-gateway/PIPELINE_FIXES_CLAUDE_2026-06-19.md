## Контекст (з аудиту)

Живий трейс показав: на запит «Телефон айфон дисплей скільки по роботі поміняти»
роут `catalog` повернув `validation_failed` → усі знайдені факти стерто → клієнт
отримав відмовку без ціни. При цьому в каталозі Є рядок «Заміна дисплейного
модуля смартфона (робота) 700-1800 грн». Тобто база і пошук працюють, а
**крок валідації знищував знайдене**.

Дві першопричини в коді:
1. `relevant:false` (обережність малої моделі) → `facts = facts if relevant else []`
   повністю обнуляв знайдене.
2. Валідатор-сесія: великий `result_validation_prompt` + до 5000 символів сирого
   результату + жорсткий ліміт `max_tokens=450` → JSON обривався → `except` →
   `validation_failed` → втрата всього.

(Окремо від коду лишається проблема стану БД: seed-міграція `06_18` могла не
залити каталог через видалений `services.json`, а `29f0 cleanup` міг вимкнути
`enabled=false`. Це перевіряється дампом БД і виправляється окремою міграцією —
НЕ в цьому журналі.)

## Правки

### FIX 1 — не викидати знайдене при `relevant:false` / збої валідації
Файл: `app/core/agent_lean.py` (`_run_route_session`, застосування фактів у циклі)
Статус: ✅ зроблено (закомічено, запушено в main 2026-06-19)

**Було:** `facts/notes/missing` передавалися лише якщо `relevant is True`; будь-який
збій парсингу → `validation_failed` з порожнім результатом.

**Стало:**
- `facts/notes/missing` передаються, якщо вони непорожні, незалежно від прапора
  `relevant`; сигнал довіри несе `match_status` (`confirmed/partial/unknown`).
- При збої JSON-парсингу — спершу толерантний порятунок (`_salvage_validation`);
  якщо й він не вдався, але інструмент внутрішнього джерела (каталог/знання/
  бізнес-інфо) щось повернув — сирий результат прокидається як
  `match_status:"unvalidated"` кандидат із застереженням, щоб модель відповіді
  бачила знайдене (напр. ціну), а не вважала, що даних нема.
- Для веб-джерел сирий результат при збої НЕ прокидається (ризик сміття) —
  лишається `validation_failed`.

**Ризик:** низький. Гірший випадок — модель відповіді бачить трохи більше
кандидатних даних із позначкою «непідтверджено», що краще за хибну відмову.

**Ризик:** низький. Більший ліміт токенів лише на внутрішньому виклику валідації.

### FIX 6 — діагностичні логи замість дампу промптів
Файл: `app/core/agent_lean.py` (`_fmt_msgs`, `_section_sizes`, `_safe_chat`)
Статус: ✅ зроблено (запушено в main)

**Було:** кожен `"Вхід у модель"` вивалював увесь системний промпт цілком —
стіна тексту, в якій не видно структури й помилок.

**Стало:**
- `_fmt_msgs` тепер показує СХЕМУ виклику: кількість повідомлень + сумарний

**Ризик:** нульовий — змінено лише вміст trace-повідомлень, не логіку пайплайна.

### FIX 8 — розпізнавання цінового наміру у fallback (2026-06-20)
Файл: `app/core/agent_lean.py` (`_fallback_route_decision`, `price_re`)
Статус: ✅ зроблено (запушено)

**Проблема (з трейсу):** контролер падає → рішення бере fallback-regex. Його
`price_re` ловив лише «скільки **коштує**», тож «скільки робота», «ціна заміни»,
«почому», просто «скільки» → класифікувались як `availability` → `scope_only`
→ каталог віддавав лише заголовки категорій, без рядків з цінами → бот не давав
ціну.

**Стало:** `price_re` ловить `ціна/ціну/ціни/вартість/скільки/за скільки/почому/
прайс/орієнтовно`. Ціновий запит → `requested_fact=price` → повний `search_catalog`
з рядками й сумами.

**Лишається окремо (системне):** DECIDE (контролер) повертає порожньо стабільно —
усе тримається на fallback. Треба з'ясувати причину (порожня відповідь моделі чи
помилка LLM на кроці DECIDE).

### FIX 9 — КОРІНЬ падіння контролера: reasoning-think з'їдав малий бюджет токенів (2026-06-20)
Файл: `app/core/agent_lean.py` (ліміти `max_tokens` стадій)
Статус: ✅ зроблено (запушено)

**Відкриття:** `llm.py:strip_think()` вирізає `<think>…</think>` (модель reasoning-
типу). Стадії з малим `max_tokens` витрачали весь бюджет на think — після нього
JSON/відповідь не встигали згенеруватись, і `strip_think` лишав `""`.

Симптоми збіглись із трейсами:
- **DECIDE** `180` → завжди порожньо → усе на fallback-regex.
- **query** `80` → завжди «Фолбек запиту».
- **conduct** `4` → завжди «normal» (ніколи не попереджав/банив).
- **validate** `450` → `validation_failed` (FIX 2 вже підняв до `800` — запрацював).
- **answer** `700-1200` → працював (бюджету вистачало).

**Стало:** DECIDE 180→**700**, query 80→**400**, conduct 4→**500**, warning 80→**400**.
Тепер після think лишається місце на корисний вивід.

**Очікування:** контролер почне повертати валідний JSON сам (fallback стане
рідкісним аварійним, а не основним шляхом); query-builder перестане падати у
фолбек; conduct реально працюватиме.

### FIX 10 — оптимізації з трейсу: дедуплікація контексту + ширші ліміти (2026-06-21)
Файл: `app/core/agent_lean.py`
Статус: ✅ зроблено (запушено)

**Підтверджено трейсом:** контролер ожив (DECIDE дає валідний JSON), ціна айфон-
дисплея дійшла (700-1800 грн). Окремо — `BadRequestError: No models loaded` —
це **LM Studio вивантажив модель на сервері**, не код (діагностика тепер це
показує явно).

**Оптимізації:**
- **Дедуплікація answer-контексту:** `facts` і повний `route_results` JSON несли
  ОДНЕ Й ТЕ САМЕ (для айфона — ~6.5k символів каталогу двічі). Прибрав
  `[ROUTE RESULTS THIS TURN]` з answer; `reply_hint` і `match_status` перенесені
  у facts. Answer-контекст помітно менший.
- **Ширші ліміти токенів** (сервер тримає 32k контексту, тож безпечно): DECIDE
  700→1000, query 80→700, validate 800→1400, answer cap 1200→2000. Менше обрізань
  валідатора → рідше salvage-шлях.

**Інфра-нота (не код):** на сервері вимкнути auto-unload моделі в LM Studio
(JIT / idle TTL) або тримати модель завантаженою — інакше серед діалогу
`No models loaded`.

### FIX 11 — людський стиль відмови + памʼять не засмічується (2026-06-21)
Файли: `app/core/prompt_defaults.py` (LEAN_ANSWER_PROMPT), `app/core/agent_lean.py`
Статус: ✅ код + answer-промпт залито міграцією `36f0` (бекап попереднього значення в meta для відкату)

**Скарга з трейсу:** на «кавомолку»→«крупс» бот зайво питав модель, а тоді все
одно відмовив фразою «у нашому переліку немає». Плюс `BUSINESS RULES` ріс 5k→8k
між повідомленнями.

**Зроблено:**
- **Памʼять (код):** `_facts` більше не зберігає сирий scope-дамп (список
  заголовків категорій) і блоби >320 символів. Саме вони накопичувались і
  роздували контекст щоходу. У поточній відповіді дані лишаються.
- **Answer-промпт (переписаний, коротший):**
  - відмова як від майстра, НЕ «немає в прайсі/базі»: тип не наш → «такого не

    беремо» (можна послатись на запчастини); клієнт наполягає на дрібниці →

  - модель пристрою питати ТІЛЬКИ коли треба шукати запчастину / зовнішню ціну,
    не для підтвердження scope чи відмови;
  - стало ~10 правил замість 20 (легше моделі, менше шаблонів).

### FIX 12 — зовнішні ціни запчастин: structured parsing + промпти + сайт (2026-06-21)

### FIX 13 — швидкість: зрізали 2 марні LLM-виклики (2026-06-21)
Файл: `app/core/agent_lean.py`
Статус: ✅ запушено

Таймери показали: кожен LLM-крок 8-33с, відповідь ~хвилина. Вузьке місце — ланцюг
викликів, не пошук. Прибрали два, що нічого не давали (код лишився, вимкнено
прапорцем — для тестів вмикається назад):
- **QUERY-LLM → off** (`meta.route_query_llm`, дефолт 0). У трейсах він завжди
  падав у «Фолбек запиту», тож запит будується детерміновано з `subject/identifier/
  operation` без LLM. −1 виклик на кожен route.
- **CONDUCT-LLM → off** (`meta.conduct_llm`, дефолт 0). Regex-precheck і так ловить
  явну лайку миттєво; LLM-модерація коштувала ~8с на КОЖНЕ чисте повідомлення. −1.

Результат: повідомлення з пошуком **5 → 3** виклики (DECIDE → VALIDATE → ANSWER),
просте **3 → 2**. Міряти тими ж `⏱`.

Наступне (узгоджено, окремим кроком): нюансовану лайку перенести в DECIDE як
прапорець `abuse` (контекстно, без окремого виклику), бан-логіка лишається в коді.
