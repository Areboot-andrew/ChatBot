# План поетапного впровадження чат-шлюзу для сервісного центру

**Мета:** єдиний шлюз між LM Studio (Gemma 3, vision + thinking) і каналами Telegram / Viber / чат сайту, з веб-адмінкою для керування знаннями, тоном, каналами та фільтрами пошуку. Деплой — Docker через Coolify у внутрішню мережу.

---

## Архітектура (цільова)

```
[Telegram] [Viber] [Чат сайту]
      \       |        /
       ▼      ▼       ▼
  ┌─────────────────────────────┐
  │  ШЛЮЗ (Docker, Coolify)     │
  │  - адаптери каналів          │
  │  - роутер намірів (intent)   │
  │  - RAG-пайплайн              │
  │  - черга запитів до моделі   │
  │  - вирізання <think>-тегів   │
  │  - ВЕБ-АДМІНКА               │
  └─────┬───────┬────────┬──────┘
        ▼       ▼        ▼
  [LM Studio] [Qdrant] [SQL-база прайсів]
   :1234/v1    вектори   точний пошук цін
        +
     [Redis] — сесії, історія діалогів, черга
```

**Ключовий принцип розділення даних:**
| Тип даних | Де зберігається | Як шукається |
|---|---|---|
| FAQ, гарантія, умови, описи послуг | Qdrant (вектори + метадані) | семантичний пошук + фільтр категорії |
| Прайси (модель → послуга → ціна) | SQLite/Postgres | точний SQL-запит за intent |
| Історія діалогу клієнта | Redis (TTL 24–72 год) | за chat_id |
| Налаштування (тон, промпт, токени) | Postgres + кеш | адмінка |

---

## Вибір шляху: Варіант А (швидкий) чи Варіант Б (повний контроль)

### Варіант А — Dify + тонкий конектор (рекомендовано для старту)
- **Dify** (self-hosted, docker-compose, деплой через Coolify) дає з коробки: веб-панель, базу знань із завантаженням файлів (PDF/DOCX/XLSX/TXT/MD), парсинг, чанкінг, метадані, налаштування системного промпта й тону, підключення LM Studio як OpenAI-сумісного провайдера, API для додатку.
- Дописуєш лише **конектор-сервіс** (~300–500 рядків Python): Telegram webhook → Dify API → відповідь; те саме для Viber і віджета сайту.
- Прайси все одно виносиш у SQL (Етап 4) — Dify викликає це через зовнішній інструмент (tool/API).
- Термін до робочого прототипу: **1–2 тижні**.

### Варіант Б — повністю власний шлюз
- FastAPI + власна адмінка (FastAPI Admin / SQLAdmin або простий фронтенд на HTMX/React).
- Повний контроль: мульти-боти Telegram через панель, своя логіка роутингу, свої фільтри.
- Термін: **4–6 тижнів**.

> Розумний компроміс: почати з Варіанта А, а конектор писати одразу так, щоб пізніше «мозок» (Dify) можна було замінити власним кодом без зміни каналів.

## ЕТАП 0. Підготовка інфраструктури (1–2 дні)

**Задачі:**
1. На VM з LM Studio увімкнути **Local Server** (OpenAI-сумісний API, порт 1234), дозволити доступ з внутрішньої мережі (Serve on Local Network).
2. Завантажити в LM Studio:
   - чат-модель: Gemma 3 (vision-варіант, якщо потрібні фото техніки);
   - embedding-модель: `multilingual-e5-large` або `bge-m3` (обов'язково багатомовна — для української).
3. Перевірити з іншої машини: `curl http://IP_LMSTUDIO:1234/v1/models`.
4. У Coolify створити проєкт, перевірити, що контейнери бачать внутрішню мережу (IP віртуалки LM Studio).
5. Зарезервувати домен/субдомен з HTTPS у Coolify (потрібен для webhook Telegram/Viber).

**Критерій готовності:** запит до `/v1/chat/completions` і `/v1/embeddings` працює з майбутнього хоста шлюзу.

---

## ЕТАП 1. Ядро шлюзу + черга (3–5 днів)

**Задачі:**
1. Сервіс на FastAPI (або деплой Dify) у Docker.
2. Клієнт до LM Studio через `openai` SDK з кастомним `base_url`.
3. **Черга запитів** (Redis): локальна модель обробляє 1 запит за раз — паралельні повідомлення стають у чергу, клієнт отримує «друкує…».
4. **Постобробка відповіді:** вирізання `<think>...</think>` (thinking-режим Gemma), обрізання markdown-артефактів, ліміт довжини.
5. Історія діалогу в Redis: ключ `dialog:{channel}:{chat_id}`, останні 10–20 повідомлень, TTL 72 год.
6. Health-check ендпоінт `/health` (для Coolify).

**Критерій готовності:** через curl можна надіслати повідомлення і отримати чисту відповідь моделі з урахуванням історії.

---

## ЕТАП 2. Веб-адмінка (5–7 днів; у Варіанті А — вже є в Dify)

**Що має керуватись через браузер:**
1. **Тон і поведінка:**
   - редактор системного промпта (тон, заборонені теми, правила ескалації);
   - шаблони привітання/прощання по кожному каналу;
   - перемикач «передати оператору при невпевненості».
2. **Канали:**
   - додавання/видалення **кількох Telegram-токенів** (кожен бот = окремий запис: токен, назва, увімкнено/вимкнено, свій промпт-доповнення);
   - токен і налаштування Viber;
   - конфіг віджета сайту: дозволені домени (CORS), колір, ім'я асистента, стартове повідомлення.
3. **База знань (RAG):**
   - завантаження файлів: PDF, DOCX, XLSX, CSV, MD, TXT;
   - вибір **категорії** при завантаженні (faq / гарантія / послуги / доставка / акції);
   - перегляд чанків, видалення документа = видалення його векторів;
   - кнопка «переіндексувати сайт» (запуск скрейпера вручну).
4. **Прайси:** окрема вкладка — завантаження xlsx, перегляд таблиці, історія оновлень.
5. **Діалоги:** перегляд логів розмов по каналах, пошук, кнопка «втрутитись оператором».
6. Авторизація адмінки (логін/пароль + IP-обмеження на внутрішню мережу).

**Критерій готовності:** зміна промпта/токена в панелі застосовується без перезапуску контейнера.

## ЕТАП 3. RAG-підсистема зі структурою та фільтрами (4–6 днів)

**Структура колекцій у Qdrant (метадані payload):**
```json
{
  "category": "warranty | faq | services | delivery | promo",
  "source": "site | upload",
  "doc_id": "uuid документа",
  "title": "назва документа/сторінки",
  "updated_at": "2026-06-09",
  "lang": "uk"
}
```

**Пайплайн інжесту:**
1. **Парсери файлів:** PDF (pdfplumber), DOCX (python-docx), XLSX/CSV (pandas → текстові рядки або в SQL, див. Етап 4), MD/TXT — як є.
2. **Скрейпер сайту:** sitemap.xml → BeautifulSoup → очищення від меню/футера → markdown. Запуск по крону (раз на добу) + вручну з адмінки.
3. **Чанкінг:** 400–600 токенів, перекриття 50–80, заголовок сторінки додається до кожного чанка (контекст).
4. **Ембедінги:** через LM Studio `/v1/embeddings`.
5. **Дедуплікація:** хеш чанка — не переіндексовувати незмінене.

**Пайплайн пошуку (на кожне питання клієнта):**
1. Класифікація питання (дешево, тією ж моделлю або правилами): ціна? гарантія? статус ремонту? загальне?
2. Якщо «ціна» → SQL (Етап 4), RAG не чіпаємо.
3. Інакше → векторний пошук у Qdrant з **фільтром по категорії** (top_k = 4–5, score_threshold ≈ 0.5).
4. Знайдені чанки + історія діалогу → системний промпт → LM Studio.
5. Якщо нічого не знайдено вище порога — чесна відповідь «уточню у майстра» + ескалація.

**Критерій готовності:** питання «яка гарантія на заміну екрана?» тягне лише чанки категорії warranty, а не весь корпус.

## ЕТАП 4. Прайси як структуровані дані (3–4 дні) — НЕ в RAG

**Чому окремо:** векторний пошук плутає схожі моделі (iPhone 13 / 13 Pro / 13 mini) і тягне в контекст зайві рядки. Точні дані = точний пошук.

**Схема БД (SQLite для старту, Postgres якщо прайсів багато):**
```sql
CREATE TABLE price_list (
  id INTEGER PRIMARY KEY,
  device_type TEXT,     -- смартфон / ноутбук / планшет / ТВ
  brand TEXT,           -- Apple / Samsung / Lenovo
  model TEXT,           -- iPhone 13, Galaxy S23
  service TEXT,         -- заміна екрана, заміна АКБ, чистка
  price_min INTEGER,
  price_max INTEGER,
  duration TEXT,        -- "1-2 дні"
  note TEXT,
  updated_at TEXT
);
```

**Потік:**
1. Адмінка приймає xlsx прайса (фіксований шаблон колонок) → pandas → валідація → запис у таблицю (повна заміна або upsert).
2. Бот при intent «ціна»: витягує з питання brand/model/service (промптом «поверни JSON») → SQL `WHERE` з нечітким збігом по model (`LIKE` + нормалізація) → у контекст моделі йдуть **тільки знайдені 1–3 рядки**.
3. Якщо знайдено кілька варіантів — бот уточнює: «У вас iPhone 13 чи 13 Pro?».
4. Якщо модель не знайдена — «орієнтовну ціну скаже майстер після діагностики, вона безкоштовна» (текст налаштовується в адмінці).

**Критерій готовності:** питання «скільки коштує поміняти батарею на самсунг с23» дає точну ціну з таблиці, у контекст не потрапляє жоден зайвий рядок прайса.

## ЕТАП 5. Канали (5–7 днів, по черзі)

**5.1 Чат сайту (перший — найпростіший):**
- JS-віджет одним `<script>`-тегом, спілкування через WebSocket (або REST + polling);
- CORS лише для домену сайту; анонімний `session_id` у localStorage;
- передача сторінки, з якої пише клієнт (контекст: він на сторінці «ремонт ноутбуків»).

**5.2 Telegram:**
- aiogram 3, режим webhook на HTTPS-домен Coolify (`/tg/{bot_id}` — окремий шлях на кожен токен з адмінки);
- підтримка фото (клієнт шле фото пошкодження → base64 → vision-модель описує і відповідає);
- мульти-боти: один контейнер обслуговує всі додані в панелі токени.

**5.3 Viber:**
- webhook через viber REST API;
- ⚠️ перевірити актуальну комерційну модель Viber для чат-ботів в Україні (з 2024 boти платні) — можливо, Viber-канал відкласти або обмежити.

**Критерій готовності:** одне й те саме питання у трьох каналах дає однакову якість відповіді; історія діалогу не змішується між каналами.

## ЕТАП 6. Якість, ескалація та прозорість (3–4 дні)

1. **Handoff на оператора:** команда/кнопка «покликати людину» + автоматична ескалація, якщо модель двічі не знайшла відповідь або клієнт роздратований. Сповіщення оператору в окремий Telegram-чат з історією діалогу.
2. **Чесність бота:** бот пише живою розмовною українською, без шаблонів — але на пряме питання «я з ботом говорю?» відповідає чесно («Я цифровий асистент сервісу, можу покликати майстра»). Це і вимога європейських норм щодо ШІ, і захист репутації: викритий обман шкодить більше, ніж чесний асистент.
3. **Гардрейли в промпті:** не вигадувати ціни, не обіцяти терміни без даних, не давати порад «розберіть самі».
4. **Тестовий прогін:** 30–50 реальних питань клієнтів (з історії листувань) → перевірка відповідей → правки промпта і бази знань.
5. Логи всіх діалогів у Postgres (для аналітики: топ питань = що додати в FAQ).

---

## ЕТАП 7. Деплой і експлуатація (2–3 дні)

1. `docker-compose.yml` для Coolify: `gateway` (+ `admin`, якщо окремо), `qdrant`, `redis`, `postgres`. Усі секрети — у змінних оточення Coolify.
2. Volumes: дані Qdrant, Postgres, завантажені файли — на персистентні томи (бекап разом з VM).
3. Моніторинг: health-check'и в Coolify, алерт у Telegram адміну, якщо LM Studio недоступний (модель «спить» / VM перезавантажилась).
4. Fallback: якщо LM Studio не відповідає 10 с — клієнту «зараз уточню, хвилинку» + ескалація оператору, а не мовчанка.
5. Оновлення прайсів — щотижня через адмінку; переіндексація сайту — крон щодня.
